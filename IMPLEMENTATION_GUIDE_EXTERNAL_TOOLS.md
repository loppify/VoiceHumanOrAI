# Посібник з інтеграції зовнішніх утиліт клонування голосу та TTS

Цей документ описує стратегію та технічну реалізацію інтеграції сторонніх інструментів (**LuxTTS**, **MOSS-TTS-Nano**, **Voicebox**, **OmniVoice-Studio**) у проєкт VoiceHumanOrAI.

---

## 1. Концепція інтеграції
Для підтримки багатьох рушіїв генерації без зміни основної логіки використовується патерн **"Стратегія" (Strategy)**. Всі генератори успадковують абстрактний клас `TTSProvider`.

### Базова архітектура (`dataset_builder.py`):
```python
class TTSProvider(ABC):
    @abstractmethod
    async def generate(self, text: str, output_path: str):
        """Згенерувати аудіо та зберегти за вказаним шляхом"""
        pass
```

---

## 2. Деталізація по інструментах

### A. LuxTTS (Найкращий для клонування)
**Призначення:** Створення ідентичних цифрових копій конкретних голосів з `dataset/human/`.
- **Метод імплементації:** Прямий імпорт Python-класу.
- **Особливість:** Потребує 48kHz референс.
- **Статус:** Імплементовано у `LuxTTSProvider`.

### B. MOSS-TTS-Nano (Локальний та швидкий)
**Призначення:** Масова генерація ШІ-зразків на CPU без сторонніх API.
- **Метод імплементації:** Виклик через `subprocess` ізольованого скрипта `infer.py`.
- **Особливість:** Потребує бібліотеки `sentencepiece` та `WeTextProcessing`.
- **Статус:** Імплементовано у `MossTTSProvider`.

### C. Voicebox (Гібридна студія)
**Призначення:** Доступ до 7 різних рушіїв (Kokoro, TADA тощо) через єдиний інтерфейс.
- **Метод імплементації:** Через REST API (якщо запущено сервер Voicebox) або CLI-shim.
- **Рекомендація:** Використовувати `curl` запити до `http://127.0.0.1:17493/generate`.

### D. OmniVoice-Studio
**Призначення:** Дослідження емоційного забарвлення.
- **Метод імплементації:** Експорт готових емоційних зразків (радість, гнів) та їх тестування на стійкість біонічного методу.

## 3. Практичне впровадження (Крок за кроком)

### Крок 1: Підготовка середовища
Оскільки зовнішні утиліти мають конфліктні залежності (наприклад, різні версії `torch` або специфічні системні ліби як `pynini`), рекомендується:
1. Використовувати основне `venv` для легких моделей (LuxTTS).
2. Використовувати `subprocess` виклики для важких або конфліктних моделей (MOSS).

### Крок 2: Оновлення `dataset_builder.py`
Додайте відповідний клас-провайдер:
```python
class NewToolProvider(TTSProvider):
    async def generate(self, text: str, output_path: str):
        # Логіка виклику тулзи
        pass
```

### Крок 3: Запуск генерації
```bash
python dataset_builder.py --provider lux --samples 50
```

## 4. Наукова цінність для проєкту
Інтеграція цих інструментів дозволяє провести **Adversarial Testing** (тестування на вразливість):
1. **Тест на схожість:** Чи відрізнить система реальну людину від її клона, створеного через LuxTTS?
2. **Тест на емоції:** Чи не прийме біонічний метод емоційну людину за ШІ через зміну стабільності частоти?
3. **Тест на якість:** Порівняння 24kHz (Edge TTS) та 48kHz (LuxTTS) синтезу.

## 5. Вирішення проблем (Troubleshooting)
1. **ModuleNotFoundError (tn, pynini):** Це частини `WeTextProcessing`. Для Linux їх найпростіше встановити через Conda або скачати готові `.whl` файли, оскільки компіляція з сирців часто потребує специфічних C++ хедерів (`fst/util.h`).
2. **CUDA Out of Memory:** При ініціалізації декількох провайдерів одночасно. Рекомендується використовувати `device='cpu'` для MOSS-TTS та `gpu` для LuxTTS.
